# Admin Panel

## Purpose

Provides a web-based administration interface for managing bots, system prompts, first messages, and viewing conversation history. Access is restricted to Telegram users listed in ADMIN_TG_IDS via one-time token authentication.

## Requirements

### Requirement: Token-Based Authentication
Admins SHALL authenticate via one-time UUID tokens generated by the /admin bot command. Tokens SHALL expire after 1 hour and be marked as used after first successful login. The login URL format is: {ADMIN_BASE_URL}/admin/login?token={token}.

#### Scenario: Admin generates login link
- **WHEN** an admin (tg_id in ADMIN_TG_IDS) sends /admin to the bot
- **THEN** a UUID token is created in admin_tokens and a login URL is returned

#### Scenario: Non-admin attempts /admin
- **WHEN** a user not in ADMIN_TG_IDS sends /admin
- **THEN** they receive an access denied message

#### Scenario: Token reuse prevention
- **WHEN** a used or expired token is submitted to /admin/login
- **THEN** the login page shows "Invalid or expired token"

### Requirement: Session Management
Authenticated sessions SHALL use express-session with ADMIN_SESSION_SECRET. Sessions SHALL store adminTgId and isAuthenticated. Session max age is ADMIN_SESSION_MAX_AGE_HOURS (default 24h). All routes except /admin/login require authentication via requireAuth middleware.

#### Scenario: Session redirects unauthorized
- **WHEN** an unauthenticated user visits /admin/prompt
- **THEN** they are redirected to /admin/login

#### Scenario: Session logout
- **WHEN** admin visits /admin/logout
- **THEN** the session is destroyed and they are redirected to login

### Requirement: System Prompt CRUD
The system SHALL support GET /api/admin/prompt to retrieve the active prompt and POST /api/admin/prompt to create a new versioned prompt. Creating a new prompt increments version and deactivates all previous prompts in a transaction.

#### Scenario: Admin updates system prompt
- **WHEN** admin submits new prompt text
- **THEN** old prompts are deactivated; new prompt with version N+1 is created; AI agent picks it up within 60s cache

### Requirement: First Message CRUD
The system SHALL support GET /api/admin/first-message and POST /api/admin/first-message. Only one first message may be active at a time (enforced by a conditional unique index).

#### Scenario: Admin updates first message
- **WHEN** admin submits new first message text
- **THEN** old message is deactivated; new message is inserted; /start command uses the new message

### Requirement: Bot CRUD
The system SHALL support: GET /api/admin/bots (list active), POST /api/admin/bots (create, token must be unique), DELETE /api/admin/bots/:id (soft-delete: is_active = false). Changes require Docker container restart to affect running bots.

#### Scenario: Admin adds a new bot
- **WHEN** admin submits a valid bot token
- **THEN** a new row is inserted into bot_configs; the UI shows a restart reminder

#### Scenario: Duplicate token rejected
- **WHEN** admin submits a token that already exists
- **THEN** a 400 error with "Bot token already exists" is returned

### Requirement: Conversation Viewer
The system SHALL support GET /api/admin/conversations with pagination and filters (tgId, dateFrom, dateTo) and GET /api/admin/conversations/:id/messages with chronological message history.

#### Scenario: Admin views conversations
- **WHEN** admin navigates to /admin/conversations
- **THEN** a paginated list of conversations is displayed with status, dates, and filters

#### Scenario: Admin views message history
- **WHEN** admin selects a conversation
- **THEN** all messages (user, assistant) are shown in chronological order

### Requirement: Web Interface
The admin panel SHALL render EJS templates with Bootstrap 5 for five views: login, system prompt editor, first message editor, bot management table, and conversation history browser.

#### Scenario: Prompt editor interface
- **WHEN** admin visits /admin/prompt
- **THEN** they see a monospace textarea with the current active prompt and a save button

### Requirement: Graceful Shutdown
The Express server SHALL be closed via server.close() on SIGTERM/SIGINT signals.

#### Scenario: Docker stop triggers graceful shutdown
- **WHEN** docker compose down is issued
- **THEN** the admin server and database connections are closed cleanly
